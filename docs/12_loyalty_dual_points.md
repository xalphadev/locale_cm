# 12 — ระบบแต้มสองสกุล (Dual-Points Loyalty): แต้มร้าน + แต้มแอป

> สถานะ: **ออกแบบเสร็จ + foundation สร้างแล้ว** (migration `0023_dual_points.sql` — apply + ผ่าน E2E บน dev DB)
> ที่มา: workflow ออกแบบ 10-agent (ground → 4 lens → adversarial 2 → synthesis) บนสคีมาจริง
> หลักการกำกับ: แต้มทั้งสองสกุล **ไม่ใช่เงิน** — แลกเป็นเงินสด/โอนกันไม่ได้ → อยู่นอกขอบเขต e-money ของแบงก์ชาติ

## 1. ภาพรวม — ทำไม "สองสกุล"

แพลตฟอร์มเดินด้วยแต้ม **2 สกุลที่แยกขาดจากกัน** และมี **ระบบ Coin/escrow เดิม** (money-backed) ที่ถูกพักไว้:

| สกุล | คือ | ใครออก/ใครจ่าย | ใช้ทำอะไร | เป็นเงินไหม |
|---|---|---|---|---|
| **แต้มร้าน (Stamps)** | บัตรสะสมแต้มดิจิทัล **ต่อแบรนด์** | ร้านออกเอง จ่ายเป็น "ของ" (in-kind) | สะสม→แลกของ/ส่วนลด/สิทธิพิเศษของร้านนั้น | ❌ ไม่ใช่ |
| **แต้มแอป (Sparks ✦)** | XP ของแพลตฟอร์ม (มีอยู่แล้ว) | แพลตฟอร์ม (+ สปอนเซอร์) | gamification ดึงคนใช้แอป → แลกของแต่งโปรไฟล์/ลุ้นรางวัล | ❌ ไม่ใช่ |
| Coin (ของเดิม) | เหรียญผูกเงินจริง 1 COIN=1฿ | ร้านวาง escrow + KYC | quest ข้ามร้านที่สปอนเซอร์จ่าย | ✅ ใช่ (พักไว้) |

**กฎเหล็กของการแยก:** เก็บคนละตาราง, คนละหน้าจอ, **ไม่มีการแปลงข้ามสกุลทุกทิศทาง**, ไม่โชว์เลข ฿ บนแต้มทั้งสอง การเชื่อมเดียวที่อนุญาตคือ **เช็คอิน 1 ครั้ง → ได้ทั้งแต้มร้าน + แต้มแอปพร้อมกัน** (trigger ร่วม ไม่ใช่ยอดร่วม)

## 2. แต้มร้าน (Stamps) — "บัตรสะสมแต้มที่ร้านคุมเอง"

**แนวคิด:** เหมือนบัตรตอกแต้มกระดาษที่ร้านแจก แต่เป็นดิจิทัล ไม่มีวันหาย เติมอัตโนมัติตอนเช็คอิน

- **Grain = ต่อแบรนด์ (ร้าน)** ไม่ใช่ต่อสาขา → ลูกค้าประจำสะสมข้ามทุกสาขาของร้านเดียวกัน (ตามโมเดล 0022 "1 เจ้าของ → หลายแบรนด์ → หลายสาขา") แต่ทุก event บันทึก `place_id` ไว้ทำ analytics รายสาขา
- **ตั้งโปรแกรมง่ายสุด — คำถามเดียว:** "ให้ **[กาแฟฟรี]** เมื่อครบ **[6]** ครั้ง" — ปุ่มอื่น (ตัวคูณ/คูลดาวน์/วันหมดอายุ) ซ่อนเป็น default ปลอดภัย เปิดเฉพาะแพ็ก Growth+
- **earn:** server เป็นคนตัดสิน (geofence) — เช็คอินที่ร้านยืนยันแล้ว (QR/OTP/ใบเสร็จ) ได้แต้ม × ตัวคูณตาม trust tier; เช็คอิน GPS เฉย ๆ ได้ 0 (กันปั๊มแต้มจากบ้าน)
- **จ่ายเป็นของ (in-kind):** ร้านมินต์ "สัญญา" ของตัวเอง (กาแฟ/ส่วนลด) → **ไม่มี escrow, ไม่มี KYC, ไม่มี take-rate**
- **catalog รางวัล (`stamp_rewards`):** `free_item` | `discount` | `privilege` (สิทธิพิเศษ — VIP/ลัดคิว/ที่นั่งสมาชิก/ของขวัญวันเกิด) คิดเป็น "แต้มของร้านนั้น" เท่านั้น
- **โปรโมชั่น (`stamp_promotions`, Growth+):** double-stamp weekend, happy-hour ตัวคูณ, "100 คนแรกรับ 30 แต้ม"
- **แลกที่เคาน์เตอร์:** ลูกค้ากดแลก → ระบบสร้างคำขอ `pending` + nonce ใช้ครั้งเดียว → แคชเชียร์ยืนยันด้วย OTP/QR (ห้าม static QR ฝั่งลูกค้า) → `fn_redeem_stamps` ตัดแต้ม (กันติดลบ + กันกดซ้ำ idempotent)
- **กันพังให้ร้านเห็น:** หน้าตั้งค่าโชว์ "≈ แจกฟรี 1 แก้ว/ลูกค้าประจำ/2 สัปดาห์; ค้างจ่าย N แก้ว" + เตือนถ้าอัตราแลก < 10% ("รางวัลยากไป ลดเหลือ 5 ครั้งไหม") + cap ต่อวันที่ปิดไม่ได้

## 3. แต้มแอป (Sparks ✦) — gamification ของแพลตฟอร์ม

ต่อยอดจาก `spark_*` ที่มีอยู่ (ปัจจุบันให้แค่ 5 แต้ม/เช็คอิน ตายตัว ไม่มีที่ใช้) — เพิ่ม **earn แบบแปรผัน** + **ที่ใช้แต้ม (SINK)** ที่ไม่เคยมี

- **earn (V1):** เช็คอินได้ Sparks แบบแปรผัน = ฐาน + **โบนัสเช็คอินแรกของวัน** + สุ่มเล็กน้อย — **server เป็นคนสุ่ม ลูกค้าแค่เห็นอนิเมชัน** (สุ่มจ่ายแค่ของไม่มีมูลค่า = ไม่ใช่การพนัน ม.8) ทุกค่าอยู่ใน `platform_config`
- **สตรีค (V1):** สตรีค **รายสัปดาห์แบบให้อภัย** (5 ใน 7 วัน ไม่ใช่ติดทุกวัน) + auto-freeze + ซ่อมได้ 48 ชม. + ไม่เคยรีเซ็ต tier/badge → retention ไม่ผูกกับสตรีคจนกดดัน
- **ที่ใช้แต้ม (SINK) — จุดที่เคยขาด:**
  - **ของแต่งโปรไฟล์ (หลัก, ต้นทุน ~0):** สกินพาสปอร์ต/อวตาร, สีตราประทับ, กรอบ, title, ลายล้านนา — **โอนให้กันไม่ได้ตามสคีมา**
  - **streak-freeze:** จ่ายแต้มกันสตรีคขาด (จำกัดต่อเดือน)
  - **(ทีหลัง, ปิดไว้ก่อน) perk จากสปอนเซอร์:** voucher TAT/ห้าง จาก pool จำกัด first-come (ไม่สุ่ม) สปอนเซอร์จ่ายของจริง
- **ทีหลัง:** league (จัดอันดับเฉพาะเช็คอินที่ยืนยัน), collectible set ตามเทศกาล (ยี่เป็ง/สงกรานต์ — สะสมครบได้สกินพิเศษ), referral (ผูกกับการที่เพื่อนเช็คอินจริง), review
- **ทำไมติดหนึบ:** เช็คอินครั้งเดียวได้ 2 สกุล (ไม่ต้องทำซ้ำ) + hook: trigger(push) → action(เช็คอิน) → reward แปรผัน(เซอร์ไพรส์) → investment(สตรีค/badge/คอลเลกชัน) → ของพวกนี้ติดตัว เอาออกไม่ได้ จึงเสียดายถ้าเลิกใช้

## 4. เช็คอิน + Analytics ที่ร้านได้เห็น (catalog)

เช็คอินในรัศมีร้าน (PostGIS geofence, server ตัดสิน) → ข้อมูลรวมแบบ **aggregate เท่านั้น + ปิดบังถ้าน้อยกว่า 5 คน** (k-anonymity, PDPA) → คีย์ด้วย `brand_id` (ไม่ใช่ `merchant_id` ที่ NULL อยู่)

| # | เมตริก | นิยาม | ทำไมร้านอยากเห็น | แพ็ก |
|---|---|---|---|---|
| **P1** | เข้าร้านที่ยืนยัน + ซ้ำ/ใหม่ | นับ verified visit/สัปดาห์ แยกหน้าใหม่-ลูกค้าซ้ำ | ตัวเลขเดียวที่คำนวณได้จากข้อมูลที่มี **วันนี้** — นำเสนอคุณค่าด้วยอันนี้ | Free/Starter |
| **P2** | ลูกค้าประจำ vs หน้าใหม่ | % กลับมาซ้ำ, เทรนด์กลุ่มประจำ (จาก `first_visit_at`) | สัญญาณว่าโปรแกรมดึง "คนกลับมา" จริงไหม (incrementality) | Growth |
| **P3** | แต้มค้างจ่าย + อัตราแลก | แต้มคงค้างทั้งแบรนด์, ออก vs แลก (30 วัน), value-at-risk | รู้ว่าค้างจ่าย "ของฟรี" เท่าไร + โปรแกรมตายหรือยัง (ขับ auto-nudge) | Starter/Growth |
| **P4** | ความถี่/recency + retention | เฉลี่ยกี่วันมาที, cohort กลับมา wk1/2/4, ลูกค้าประจำที่หายไป | ยิง win-back, จัดกะ, จูน threshold | Growth |
| **P5** | ฮีตแมป ช่วงเวลา×วัน | เช็คอินเกิดตอนไหน (ชม.×วัน) | ทำโปรเองได้ ("อังคารบ่ายเงียบ → double-stamp happy hour") | Starter/Growth |
| **P6** | ประสิทธิภาพรางวัล/โปร | แลกต่อรางวัล, เวลาแลกครั้งแรก, ยอดเพิ่มช่วงโปร | ตัดรางวัลที่ตาย, ตั้งราคาแต้มถูก, พิสูจน์ว่าโปรแกรมคุ้ม (เหตุผลต่ออายุ) | Growth |
| **P7** | สัดส่วนกลุ่มลูกค้า | local / nomad / tourist_west / tourist_cn (ปิดบังถ้า n<5) | คาเฟ่นิมมานรู้ว่า 60% จีน → เมนู zh + Alipay; ร้านท้องถิ่นรู้ว่า 90% local | analytics_pro |
| **P8** | คุณภาพการมา (dwell) | median dwell, สัดส่วน proven_purchase vs ผ่าน | ดูว่าเป็น footfall จริงหรือผ่านเฉย ๆ | analytics_pro |
| **P9** | catchment + ลูกค้าร่วมร้าน | เขตที่อยู่คร่าว ๆ ของลูกค้าประจำ (k≥25), ร้านอื่นที่ลูกค้าทับกัน | insight ลึกสุด ขายเป็น data-product | Pro/data |

> ตอนนี้สร้างได้จริง = P1 (จาก `check_ins` ที่มี), P3 (จากแต้ม) ส่วน P2/P7 ต้องรอข้อมูลสะสม + เขียน writer ของ `audience_segment` → หน้า dashboard ที่ยังไม่มีข้อมูลให้ขึ้น "ดูได้เมื่อมีข้อมูลพอ"

## 5. Business Model

- **ใครจ่ายอะไร:** แต้มร้าน = ร้านจ่ายเป็นของ 100% (แพลตฟอร์มไม่ตั้งสำรอง/escrow); Sparks หลัก = แพลตฟอร์มจ่ายแต่ต้นทุน ~0 (ดิจิทัล); perk จริง = สปอนเซอร์จ่ายจาก pool จำกัด
- **รายได้ 4 ขา:**
  1. **ค่าสมาชิกร้าน (หลัก):** Free 0฿ (โปรไฟล์+เคลมร้าน ไม่มีโปรแกรม) → **Starter 390฿** (แต้มพื้นฐาน 1 รางวัล + P1/P3/P5) → **Growth 990฿** (catalog เต็ม + สิทธิพิเศษ + โปรโมชั่น + CRM + P2/P4/P6/P7) → **Pro 1,690฿** (แต้มข้ามสาขา + P8/P9 + API) + add-on `analytics_pro`
  2. **แคมเปญ Sparks สปอนเซอร์:** ห้าง/แบรนด์/ททท. จ่ายโบนัส Sparks + pool perk (มาร์จิ้นสูง — Sparks ต้นทุน ~0)
  3. **featured placement / โปรโมชั่นเด่น** (ติดป้าย "Sponsored")
  4. **data product (k≥25)** + affiliate booking (Klook ฯลฯ)
- **ไม่เก็บ:** **ไม่มี take-rate บนการแลกแต้ม** — take-rate บนแต้ม = settlement = ดึง e-money/escrow กลับมา (take-rate 10% เหลือไว้เฉพาะ Coin)
- **ความคุ้มทุนดีกว่า Coin มาก:** ปัญหา break-even ~46% / incrementality 10-30% ในเอกสารเดิมเป็นของโมเดล Coin (รางวัลเป็นเงินจริงที่ร้านต้องวางล่วงหน้า) — แต้มร้านตัด escrow-COGS ทิ้ง ร้านแค่ให้ของต้นทุนชายขอบ (กาแฟต้นทุน ~12฿) เฉพาะลูกค้าที่กลับมาแล้วเท่านั้น → P&L คุมด้วยค่าสมาชิกคงที่ ไม่ใช่เงินไหลออกแปรผัน

## 6. การตัดสินใจเรื่องระบบ Coin/escrow เดิม → **อยู่ร่วม แต่พักไว้**

ไม่ลบ migration 0001-0012 (ledger/escrow/FIFO/take-rate ถูกต้องและเป็น moat) — แต่ **พักการใช้งานประจำวัน**:
- เก็บ Coin ไว้สำหรับ niche เดียว: **quest ข้ามร้านที่สปอนเซอร์/เทศกาลจ่าย** (ห้าง/แบรนด์/ททท.) + ร้านใหญ่/เชนที่ยอมทำ KYC+prefund (opt-in, Pro+)
- ลอยัลตี้ประจำวัน = แต้มร้าน + Sparks (ไม่แตะ ledger/escrow/currency enum → ไม่ติด KYC/PSP/BoT)
- **ไม่มีการแปลงข้ามสกุล** ทุกทิศทาง
- ปัจจุบัน Coin **inert อยู่แล้ว** (COGS cap = NULL → `fn_grant_coins` fail-closed; ทุกร้าน `merchant_id` = NULL ไม่เคย transact) → พักไว้ = ไม่มีอะไร regress

## 7. Anti-fraud + กฎหมาย

- **ไม่ใช่เงินโดยโครงสร้าง (ไม่ใช่นโยบาย):** แลกเงินสดไม่ได้ + โอนกันไม่ได้ (ไม่มีคอลัมน์/endpoint โอน) + แปลงข้ามสกุลไม่ได้ + แต้มร้านจ่ายเป็นของ → อยู่นอกนิยาม e-money. บังคับด้วย `CHECK(balance>=0)`, `is_transferable=false`, ไม่เพิ่มค่าใน `currency` enum, ไม่โพสต์ลง `ledger_entries`
- **เส้น ม.8 (พนัน):** สุ่มจ่ายแค่ Sparks/ของแต่ง (มูลค่า 0); ของแต่งห้ามกลายเป็น input ปลดล็อกของมีมูลค่า (transitivity); perk มีมูลค่าจริงที่สุ่ม = ต้องมีใบอนุญาต ม.8 (เก็บ ref ใน DB) ปิดไว้ก่อน — sink ที่ใช้คือ first-come จาก pool จำกัด (ไม่สุ่ม)
- **anti-farm (โค้ดจริง):** geofence server-side (`ST_DWithin`) + cap ต่อ (user,brand,วัน) + device dedupe + impossible-travel + ถ่วงน้ำหนัก trust tier (GPS เฉย ๆ ได้ 0) + z-score → ตั้ง flag "กำลังตรวจสอบ" (ไม่ shadow-ban เงียบ)
- **PDPA:** aggregate-only, ปิดบัง n<5, geo คร่าว+ลบทิ้งตามรอบ, ไม่เก็บกราฟการเคลื่อนที่ — เก็บแค่ "ข้อเท็จจริงว่ามาเช็คอิน"

## 8. สิ่งที่สร้างแล้ว (migration `0023_dual_points.sql`) — ✅ ผ่าน E2E

**ตาราง:** `stamp_programs`, `stamp_balances` (floor ≥0), `stamp_events` (append-only, trigger immutable), `stamp_rewards`, `stamp_promotions`, `shop_redemptions`, `checkin_daily_caps`, `platform_rewards`, `spark_redemptions`, `user_cosmetics` (โอนไม่ได้) + `spark_balances` เพิ่ม `CHECK(balance>=0)`

**ฟังก์ชัน:** `award_shop_points` (earn/burn floored), `fn_redeem_stamps` (fail-closed + idempotent + quota), `spend_sparks` (floored debit + idempotent + cosmetic), `fn_shop_checkin` (geofence → check_in → แต้มร้าน + Sparks fan-out), `cfg_num` (อ่าน config)

**ตรวจแล้ว (E2E, in-tx rollback):** สะสม 6 แต้ม → first_visit_at set → เช็คอินจ่าย 2 สกุล → แลกของ (6→0, idempotent) → ติดลบถูกปฏิเสธ → จ่าย Sparks ได้ของแต่ง (idempotent) → โอนไม่ได้ถูก block

> หมายเหตุ: `fn_shop_checkin` ใช้ PostGIS (`ST_DWithin`) — dev DB stub PostGIS จึงทดสอบ logic แต้มผ่าน primitives โดยตรง (เหมือน `fn_check_in` เดิมที่ทดสอบบน PostGIS จริงเท่านั้น)

## 9. Roadmap การสร้าง (ที่เหลือ)

- **Phase 0 — เคลม/ออนบอร์ด (บล็อก):** ร้านเคลม place → สร้าง merchant_account + brand → ปลดล็อกสร้างโปรแกรม (ตอนนี้ live places ส่วนหนึ่ง brand_id ยังว่าง)
- **Phase 1 — vertical slice (ใช้งานครบวงบนร้านเดียว):** wire `fn_shop_checkin` แทน check-in เดิมที่ไม่มี geofence → ปุ่ม "เช็คอินที่นี่" ฝั่งลูกค้า → หน้าตั้งโปรแกรม "คำถามเดียว" ฝั่งร้าน → catalog + แลกที่เคาน์เตอร์ → wallet "บัตรร้านของฉัน" + Sparks header
- **Phase 2 — Sparks sink + สตรีค:** ร้านค้าของแต่ง + spend_sparks UI + สตรีค 5/7
- **Phase 3 — analytics จริง:** view P1/P3/P5 (ปิดบัง n<5) + เขียน audience_segment + purge geo cron
- **Phase 4 — monetization:** tier-gating + โปรโมชั่น Growth+ + analytics_pro
- **Phase 5 — tourist loop + engagement:** collectible เช็คอินแรก + passport ข้ามร้าน + league/festival set
- **Phase 6 — Coin niche (opt-in):** quest สปอนเซอร์ข้ามร้าน (ต้องตั้ง COGS cap)

## 10. การตัดสินใจที่ต้องให้ founder (พร้อม default ที่แนะนำ)

1. **Grain แต้ม — ต่อแบรนด์ vs ต่อสาขา** → **default: ต่อแบรนด์** (สะสมข้ามสาขา ตรงกับวิสัยทัศน์ "แต้มของร้าน") — baked ใน schema แล้ว ยืนยันว่าแฟรนไชส์จะไม่ขอแยกรายสาขาทีหลัง
2. **Free มีโปรแกรมไหม** → **default: Free = โปรไฟล์+เคลมเท่านั้น**, โปรแกรมเริ่มที่ Starter 390฿ (โปรแกรมคือ upsell)
3. **ครบกี่ครั้งได้รางวัล (default)** → **default: 6 ครั้ง** + nudge ถ้าแลก<10% + ขั้นต่ำ 3
4. **Sparks — global vs รายเมือง** → **default: global** (เมืองเดียวตอนนี้) ตัดสินก่อนเปิดใช้ spend
5. **perk มีมูลค่าจริง — เปิดเลยไหม** → **default: V1 มีแค่ของแต่ง+streak-freeze (มูลค่า 0)**; perk สปอนเซอร์ปิดไว้จนผ่านรีวิว ม.8
6. **COGS cap ของ Coin niche** → **default: ปล่อย NULL (Coin พักต่อ)** จนกว่าจะได้ดีล quest สปอนเซอร์จริง

## 11. ความเสี่ยงที่ต้องระวัง (สรุป)

1. **(บล็อก) check-in เดิมไม่มี geofence + ผูก quest** → ต้องเปลี่ยนมาใช้ `fn_shop_checkin` ก่อน ไม่งั้นปั๊มแต้มจากบ้านได้
2. **(บล็อก) cold-start:** live places บางส่วน brand_id ว่าง → ต้องมี flow เคลมก่อน; ร้านที่ยังไม่เข้าร่วม เช็คอินได้แค่ Sparks
3. **analytics ขายก่อนมีข้อมูล:** P2/P7 ต้องรอสะสม + เขียน writer → นำด้วย P1 ที่มีจริง อย่าออกใบเสร็จคุณค่าที่ยังให้ไม่ได้
4. **ร้านตั้งรางวัลถูกไป → แจกฟรีเลือดไหล** (โทษแพลตฟอร์ม) → UI คำถามเดียว + default ปลอดภัย + panel ค้างจ่าย
5. **รางวัลยากไป → โปรแกรมตาย** → auto-nudge ถ้าแลก<10%
6. **non-cashability ต้องบังคับเชิงโครงสร้าง** ไม่ใช่สัญญา — ฟีเจอร์ "ส่งแต้มให้เพื่อน"/"แปลงแต้ม" ต้องผ่านที่ปรึกษากฎหมายก่อน
7. **คนละ cohort:** บัตร 6 ครั้ง + สตรีครายสัปดาห์ ไม่เหมาะนักท่องเที่ยว 5 วัน → ทำ loop นักท่องเที่ยวแยก (collectible/passport ข้ามร้าน)
