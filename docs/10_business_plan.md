## 1. สรุปสั้น (Executive Summary)

**Soi Hop คืออะไร:** แอปท่องเที่ยว–ไลฟ์สไตล์ระดับ *ท้องถิ่นเชียงใหม่* ที่บอกว่า "กิน–เที่ยว–ทำอะไรรอบตัว" พร้อม **เกมสะสมแสตมป์ข้ามร้าน** (Quest) — ไปคาเฟ่ A + คาเฟ่ B + ร้านอาหาร C ครบ แล้วได้รางวัล เช่นกาแฟฟรี/ส่วนลด — บวกชุมชนรีวิวคนเชียงใหม่จริง รองรับ 3 ภาษา (ไทย/อังกฤษ/จีน)

**ปัญหาที่แก้:**
- **ฝั่งนักท่องเที่ยว/คนเมือง:** ข้อมูลร้านในเชียงใหม่กระจัดกระจาย ล้าสมัย (ร้านปิดไปแล้วยังขึ้น) และไม่มีตัวช่วยพา "ออกไปลองร้านใหม่"
- **ฝั่งร้านเล็ก/ร้านอินดี้:** ถูกบีบโดย Wongnai/Grab (จ่ายค่าโฆษณา/ค่า GP แพง) ไม่มีเครื่องมือ *ดึงคนเดินเข้าร้านจริง* และวัดผลได้ในงบที่คุมได้

**ทำไมถึงน่าทำ (จุดต่างที่ลอกยาก):**
1. **เครือข่าย Quest ข้ามร้าน** — ทำให้ร้านเล็กหลายร้าน "รวมหัว" ดึง footfall ให้กัน (ร้านเดียวทำไม่ได้, แพลตฟอร์มใหญ่ไม่อยากทำเพราะมันไม่ scale แบบกดปุ่ม)
2. **ข้อมูลสด ตรวจสอบโดยคนจริง** — ใช้ "Field Agent" (นักศึกษา มช./พายัพ) ลงพื้นที่ตรวจ/อัปเดตร้าน แต่ agent *ไม่เคยเขียนข้อมูลสด* เขียนได้แค่ "คำขอแก้ไข" ที่ต้องผ่านการอนุมัติ → ข้อมูลสดกว่าคู่แข่ง
3. **3 ภาษาตั้งแต่วันแรก** — จับทั้งคนไทย/นักท่องเที่ยวฝรั่ง/นักท่องเที่ยวจีน FIT (เที่ยวเอง)

**คาถาเชิงกลยุทธ์:** *"บริษัทคือเครือข่ายร้านค้า ไม่ใช่ตัวแอป"* — คูเมืองอยู่ที่ความสัมพันธ์กับร้านและความสดของข้อมูล ไม่ใช่ฟีเจอร์ที่ลอกได้ใน 1 สัปดาห์

**โมเดลรายได้ 4 ขา:** (1) ค่าสมาชิกร้าน (SaaS รายเดือน) · (2) ส่วนแบ่งตอนแลกรางวัล (take-rate) · (3) Quest สปอนเซอร์ + แคมเปญเทศกาล · (4) ขายข้อมูลเชิงสถิติ (ไม่ระบุตัวตน) + ค่าคอมจองทัวร์/ที่พัก (Klook/TakeMeTour)

**เป้าแรก:** ปักหมุดที่ย่าน **นิมมาน** ให้แน่นก่อน (ร้าน ~คาเฟ่/อาหาร/สถานที่ หนาแน่นพอจน "แผนที่ไม่ว่าง") → ขยายทีละย่าน → ทั้งเชียงใหม่ → จังหวัดอื่น

## 2. ปัญหา & โอกาสตลาด

### 2.1 ฝั่งผู้บริโภค (คนเที่ยว/คนอยู่เชียงใหม่)
- เชียงใหม่เป็นจุดหมายท่องเที่ยวอันดับต้นของไทย + เป็นเมือง **digital nomad** ระดับโลก + มีนักท่องเที่ยวจีนแบบ FIT (เที่ยวเอง ไม่ทัวร์) จำนวนมาก
- ข้อมูลร้านอยู่กระจายใน Google Maps / กลุ่ม Facebook / Wongnai — ปัญหาคือ **ความสด** (ร้านย้าย/ปิด/เปลี่ยนเวลา) และ **ไม่มีแรงจูงใจให้ลองร้านใหม่**
- ไม่มีแอปที่ทำให้ "การออกไปเดินเที่ยวย่าน" สนุกเป็นเกมและเชื่อมกับคนท้องถิ่น

### 2.2 ฝั่งร้านค้า (โดยเฉพาะร้านอินดี้)
- ร้านเล็กแข่งกับโฆษณาบนแพลตฟอร์มใหญ่ไม่ไหว และค่า GP เดลิเวอรี่กินกำไร
- สิ่งที่ร้านอยากได้จริงคือ **"คนเดินเข้าร้านเพิ่ม"** ที่วัดผลได้และคุมงบได้ ไม่ใช่ยอด engagement ลอยๆ
- ร้านไม่มีเครื่องมือทำ loyalty ข้ามร้าน (จับมือร้านข้างๆ ดึงลูกค้าหมุนเวียน)

### 2.3 ช่องว่างที่ "เจ้าใหญ่ไม่เล่น"
- Wongnai/Grab = duopoly ปิดในตลาด *อาหาร/เดลิเวอรี่* → **เราจะไม่ไปแข่งตรงนั้น**
- งานที่ใช้คนลงพื้นที่ + ดีลกับร้านเล็กทีละร้าน + เครือข่าย quest ข้ามร้าน เป็นงาน "หนัก ไม่เซ็กซี่ ไม่ scale ด้วยปุ่มเดียว" → เจ้าใหญ่ไม่อยากทำ = ช่องของเรา

## 3. ผลิตภัณฑ์: คืออะไร ทำอะไรได้บ้าง

Soi Hop มี **3 ฝั่งผู้ใช้** (และระบบหลังบ้านเชื่อมทั้งหมด):

### 3.1 ฝั่งลูกค้า (Consumer app) — "กิน เที่ยว ทำอะไรรอบตัว + ชุมชน"
สร้างเป็นเว็บแอปรันได้จริงแล้ว (โปรดักชันจริงจะเป็น Flutter):
- **สำรวจ (Discover):** ฟีดชุมชน — ค้นหาร้าน, หมวด กิน/เที่ยว/ทำ, การ์ดร้านพร้อมรูปจริง+เรตติ้ง, รีวิวล่าสุดจากชุมชน, อีเวนต์/เทศกาลใกล้ๆ
- **แผนที่ (Map):** เปิดแผนที่ดูร้านรอบตัวจากตำแหน่งปัจจุบัน, กรองตามหมวด, กดหมุดดูข้อมูล + ระยะเดิน, รายการแนะนำเรียงตามระยะใกล้
- **รายละเอียดร้าน:** รูป, เรตติ้ง, เวลาเปิด-ปิด (บอก "เปิดอยู่ตอนนี้"), ที่อยู่/โทร/ไลน์/ลิงก์ Google Maps, รีวิว, quest/อีเวนต์ที่เกี่ยวข้อง, ปุ่มบันทึก (saved places)
- **Passport (สะสมแสตมป์):** ทำ Quest — เช็กอินครบร้านในเควสต์ → ปลดล็อกรางวัล
- **กระเป๋า (Wallet):** **Sparks** (แต้ม XP ฟรี ไว้เล่นเกม/แข่ง leaderboard — แลกเป็นเงินไม่ได้) กับ **Coins** (มูลค่าจริงที่ร้านเติมไว้ล่วงหน้า — แสดงเป็น "จำนวน/รางวัล" ไม่เคยโชว์เป็นบาท เพื่อเลี่ยงสถานะ e-money)
- **โปรไฟล์ & ชุมชน:** โปรไฟล์ (แต้ม/รีวิว/ที่บันทึก/เควสต์), หน้าชุมชน (top reviewers + ฟีดรีวิว) = **ชั้นสังคม** ที่ทำให้คนกลับมา

### 3.2 ฝั่งร้านค้า (Merchant + Counter)
- สมัคร/ยืนยันตัวตนร้าน (ยืนยัน 2 ชั้น: ตัวตน → การเงิน, คนตรวจคนละคน เพื่อกันโกง)
- **เติมเงิน escrow:** ร้าน/สปอนเซอร์เติมเงินบาทเข้ากระเป๋าล่วงหน้า → ใช้เป็นทุนรางวัล (แพลตฟอร์ม **ไม่เคยสำรองจ่ายแทน**; ยอดหมด → เควสต์พักอัตโนมัติ) → ความเสี่ยงทางการเงินของแพลตฟอร์ม = 0
- **Counter (จุดแลกรางวัล):** ตอนลูกค้าแลก ร้านเป็นคน "ยืนยันแลก" เอง (ไม่ใช่ลูกค้ากดเบิร์นเอง) ผ่าน QR/OTP หมุน + geofence → กันโกง
- เห็นผลลัพธ์: คนเดินเข้าร้านจาก quest, ยอดแลก, งบที่เหลือ

### 3.3 ฝั่ง Field Agent (นักศึกษาลงพื้นที่)
- รับงานสำรวจ/อัปเดตร้าน, ถ่ายรูป, เสนอ "คำขอแก้ไขข้อมูล" (change-proposal) — **ไม่เขียนข้อมูลสดเอง** ต้องผ่านโมเดอเรเตอร์อนุมัติ (คนเสนอ ≠ คนอนุมัติ)
- ได้ค่าตอบแทนแบบ task-bounty + คอมมิชชันจากการชวนร้านสมัคร

### 3.4 ฝั่งแอดมิน/หลังบ้าน
- คิวอนุมัติคำขอแก้ไขข้อมูล (ดู diff + EXIF/พิกัด/รูป), ตรวจร้าน, จัดการรีวิว, จ่ายเงินร้าน (payout + หักภาษี), แดชบอร์ดความมั่นคงทางบัญชี (solvency), ระบบ freeze ฉุกเฉิน

> **กลไกหัวใจ (Quest loop):** ร้านเติม escrow → สร้าง quest → ลูกค้าเช็กอินครบ → ระบบ mint Coins (ค้ำด้วยเงินจริงในกระเป๋าร้าน) → ลูกค้าเอา Coins ไปแลกที่ร้าน *พาร์ตเนอร์* (ไม่ใช่ร้านที่ออกทุน — กันการปั่นแลกตัวเอง) → ร้านพาร์ตเนอร์ได้ยอด, แพลตฟอร์มได้ take-rate

## 4. กลุ่มลูกค้า / ตลาดเป้าหมาย

มีลูกค้า **2 ด้านที่ต้องหาให้สมดุล** (two-sided marketplace): **ผู้ใช้** กับ **ร้านค้า** — และตัวเร่งคือ **Field Agent**

### 4.1 ผู้ใช้ (Demand) — 3 กลุ่มหลัก ทุกกลุ่มเป็นพลเมืองชั้นหนึ่ง
| กลุ่ม | ใครคือ | เข้าถึงผ่าน | จ่ายเงินผ่าน |
|---|---|---|---|
| คนไทย + digital nomad ที่อยู่ยาว | คนเชียงใหม่/คนไทยมาอยู่/ฝรั่งอยู่ยาว | LINE login, ชุมชน, รีวิว | PromptPay |
| นักท่องเที่ยวฝรั่ง | สาย backpack/คาเฟ่/วัฒนธรรม | โรงแรม/โฮสเทล, Google, Klook | บัตร/PromptPay |
| นักท่องเที่ยวจีน FIT (เที่ยวเอง) | กลุ่มโตเร็ว ใช้จ่ายสูง | ภาษาจีนในแอป, ช่องทางจีน | Alipay/WeChat |

### 4.2 ร้านค้า (Supply) — ลำดับความสำคัญ
1. **ร้านอินดี้หนาแน่นในย่านเดียว** (คาเฟ่/ร้านอาหาร/ของหวาน ในนิมมาน) — กลุ่มแรกที่ต้องได้ เพราะ quest ข้ามร้านต้องมี "ร้านข้างๆ" พอ
2. ร้านสาย "เที่ยว/ทำกิจกรรม" (สปา, มวยไทย, คลาสทำอาหาร, แกลเลอรี) — ได้กำไรต่อหัวสูง อยากได้นักท่องเที่ยว
3. สปอนเซอร์/แบรนด์/งานเทศกาล — ซื้อ quest แคมเปญ

## 5. โมเดลรายได้ (ทำเงินอย่างไร — 4 ขา)

> รายละเอียดบัญชี/ราคา canonical ดู [02_canonical_ledger_pricing_legal.md](02_canonical_ledger_pricing_legal.md)

### ขาที่ 1 — ค่าสมาชิกร้าน (SaaS subscription) · *รายได้ที่คาดเดาได้*
| แพ็กเกจ | ราคา/เดือน | สำหรับ |
|---|---|---|
| Free | 0฿ | ร้านลองใช้ มีโปรไฟล์พื้นฐาน |
| Starter | 390฿ | ร้านเล็กเริ่มทำ quest |
| Growth | 990฿ | ร้านที่ใช้จริงจัง + analytics |
| Pro | 1,690฿ | ร้านใหญ่/เชน + ฟีเจอร์เต็ม |
| + add-on `analytics_pro` | เสริม | ร้านที่อยากได้ข้อมูลลึก |

### ขาที่ 2 — ส่วนแบ่งตอนแลกรางวัล (Redemption take-rate)
- เก็บ ~**6% (อัตรามีผลขั้นต่ำ)** จากมูลค่ารางวัลที่ถูกแลกจริง (เพดานที่ออกแบบไว้ ~10–15%)
- **รางวัลมูลค่าต่ำกว่า 30฿ = ฟรี** (ไม่เก็บ) เพื่อให้ร้านกล้าตั้งรางวัลเล็กๆ ดึงคน
- จุดสำคัญ: เก็บ "ตอนเกิดผลจริง" (มีคนเดินเข้าร้านมาแลก) ไม่ใช่เก็บค่าโฆษณาลม

### ขาที่ 3 — Quest สปอนเซอร์ + แคมเปญเทศกาล
- แบรนด์/ห้าง/ททท./งานเทศกาล (เช่น ยี่เป็ง, สงกรานต์) ซื้อ quest ธีมพิเศษ, รางวัลใหญ่, exposure
- มาร์จิ้นสูง ขายเป็นแคมเปญ

### ขาที่ 4 — ข้อมูล (ไม่ระบุตัวตน) + ค่าคอมจอง (Affiliate)
- ขายรายงานเทรนด์ย่าน/footfall เชิงสถิติ (ไม่ระบุตัวตน, ภายใต้ PDPA) ให้ร้าน/อสังหา/แบรนด์
- ค่าคอมจองทัวร์/ที่พัก/กิจกรรมผ่าน Klook/TakeMeTour (ลูกค้าค้นเจอกิจกรรมในแอป → จองต่อ)

> **⚠️ เศรษฐศาสตร์ที่ต้องระวัง (สำคัญมาก):** ดีลรางวัลของร้านจะ "คุ้ม" ก็ต่อเมื่อ quest สร้าง *ลูกค้าใหม่จริง* (incrementality) ~**46% ขึ้นไป** — ถ้าต่ำกว่านั้น **ร้านขาดทุน** (เท่ากับแจกส่วนลดให้คนที่จะมาอยู่แล้ว) เราล็อกความเสี่ยงนี้ด้วย **เพดานต้นทุนรางวัลต่อครั้ง + เพดานงบ/เดือนต่อร้าน** และต้องวัด incrementality จริงก่อนตั้งราคาตอนเปิดเต็ม ไม่งั้นยิ่งโต = ยิ่งทำให้ร้านขาดทุน = ร้านเลิกใช้

## 6. คูเมือง — ทำไมคู่แข่งลอกยาก (Moat)

1. **เครือข่าย Quest ข้ามร้าน** = network effect — ยิ่งร้านในย่านมาก quest ยิ่งน่าเล่น, ผู้ใช้ยิ่งมา, ร้านยิ่งอยากเข้า (วงจรเสริมกันเอง) คู่แข่งที่มาทีหลังต้องสร้างความหนาแน่นใหม่ทั้งย่าน
2. **ข้อมูลสด ตรวจโดยคน** (Field Agent + กระบวนการอนุมัติ) — ความสดเป็นของที่ "ต้องลงแรงต่อเนื่อง" ลอกด้วยโค้ดไม่ได้
3. **ความสัมพันธ์กับร้าน** — "บริษัทคือเครือข่ายร้าน ไม่ใช่แอป" — ดีล, ความไว้ใจ, escrow ที่ร้านเติมไว้ = switching cost
4. **3 ภาษา + ช่องจ่ายเงินครบ** (LINE/PromptPay/Alipay/WeChat) — เข้าถึงจีน FIT ที่เจ้าอื่นในพื้นที่ทำไม่ครบ

## 7. แนวทางหาลูกค้า (Customer Acquisition) — หัวใจของคำถาม

ปัญหาคลาสสิกของ marketplace คือ **"ไก่กับไข่"**: ผู้ใช้ไม่มาถ้าไม่มีร้าน, ร้านไม่จ่ายถ้าไม่มีผู้ใช้ — วิธีแก้ของเราคือ **เริ่มจาก supply ที่หนาแน่นในพื้นที่เล็กมากก่อน (นิมมาน)** แล้วใช้ quest เป็นทั้งตัวดึงผู้ใช้และตัวพิสูจน์คุณค่าให้ร้าน

### 7.1 หา "ร้านค้า" ก่อน (Supply-first) — งานนี้คือธุรกิจตัวจริง
**กลยุทธ์: ปูพรมทีละย่าน (district-by-district), เริ่มที่นิมมาน**
- เลือก 1 ย่านที่ร้านหนาแน่น (นิมมาน) → ปิดดีลให้ได้ **มวลวิกฤต ~20–40 ร้าน** ในรัศมีเดินถึงกัน ก่อนเปิดให้ผู้ใช้ → "แผนที่ไม่ว่าง" ตั้งแต่วันแรก
- **Founder-led sales:** ช่วงแรกผู้ก่อตั้งเดินคุยร้านเอง (ไม่จ้างเซลส์) เพื่อเข้าใจว่าร้าน "ยอมจ่ายเพื่ออะไร" จริงๆ
- **ข้อเสนอที่ทำให้ร้าน "ตอบรับ" ง่าย:**
  - เริ่มฟรี (Free tier) — ความเสี่ยง 0 บาทสำหรับร้าน
  - ร้านคุมงบเอง (เติม escrow เท่าที่อยากเสี่ยง, ยอดหมด=พักเอง)
  - ขายด้วย "คนเดินเข้าร้านที่วัดผลได้" ไม่ใช่ "ยอด engagement"
  - quest ข้ามร้าน = ร้านได้ลูกค้าจาก *ร้านเพื่อนบ้าน* (เล่าเป็นเรื่อง "เรารวมหัวดึงคนเข้าย่าน")
- **ใช้ Field Agent เป็นทีมขายภาคพื้น:** นักศึกษา มช./พายัพ ลงพื้นที่ทั้งเก็บข้อมูลร้าน *และ* ชวนร้านสมัคร — ได้ค่าคอมเมื่อร้านสมัคร → ขยาย supply ด้วยต้นทุนผันแปร ไม่ใช่เงินเดือนประจำ
- **เป้าหมายเชิงตัวเลขก่อนเปิดผู้ใช้:** ความหนาแน่นร้านพอที่ quest 1 อันมีร้านให้เดินครบได้จริงในย่านเดียว (Gate "Seed Dense" ใน [09_build_roadmap.md](09_build_roadmap.md))

### 7.2 หา "ผู้ใช้" (Demand) — ดึงเข้ามาหลัง supply พร้อม
- **Content-led / ชุมชนรีวิว:** seed รีวิว+ร้านจริงให้แอปดู "มีชีวิต" ตั้งแต่เปิด (ทำแล้วในเดโม่: 16 ร้าน + 36 รีวิว) → คนเปิดมาเจอของจริง ไม่ใช่หน้าว่าง
- **Quest = viral loop:** เล่นเป็นกลุ่ม/ชวนเพื่อน, แชร์แสตมป์/รางวัล → ผู้ใช้พาผู้ใช้ (ระวังเรื่อง referral ที่เปิดช่อง Sybil — คุมด้วย fraud-gate)
- **LINE:** คนไทยล็อกอิน/รับข่าวสารผ่าน LINE = ช่องถูกและคุ้นเคย
- **ช่องนักท่องเที่ยว:**
  - ฝรั่ง: ดีลกับ **โรงแรม/โฮสเทล/คาเฟ่ co-working** วาง QR "ค้นพบนิมมานแบบเล่นเกม", ลิงก์จาก Klook/TakeMeTour
  - จีน FIT: คอนเทนต์ภาษาจีน + ช่องทางจีน (ระวังข้อกฎหมายแพลตฟอร์มจีน) + รับ Alipay/WeChat ในแอป
- **อีเวนต์/เทศกาล:** ใช้ช่วงเทศกาล (ยี่เป็ง/สงกรานต์/ตลาดนัด) เป็น quest แคมเปญดึงผู้ใช้พุ่งเป็นช่วง

### 7.3 หา "Field Agent" (ตัวเร่งทั้งสองฝั่ง)
- รับสมัครนักศึกษา มช./พายัพ เป็น freelancer (ค่าตอบแทน task-bounty + คอมร้าน, หัก ณ ที่จ่าย 3% เป็นค่าเริ่มต้น)
- เป็นทั้งทีมเก็บข้อมูล (ทำให้ข้อมูลสด = คุณค่าผู้ใช้) และทีมขายร้าน (ขยาย supply) ในคนกลุ่มเดียว — ต้นทุนต่ำ, เข้าถึงพื้นที่จริง

### 7.4 ลำดับการเปิด (Acquisition sequencing)
```
1) ปิดดีลร้านในนิมมานให้หนาแน่น (supply)  →
2) seed ข้อมูล+รีวิว ให้แอปดูมีชีวิต        →
3) เปิดผู้ใช้ในย่านนั้น + quest ตัวแรก       →
4) วัด incrementality จริง (quest พาคนใหม่จริงไหม) →
5) ถ้าผ่าน Gate-B (~46%) → ขยายย่านถัดไป     →
6) ทำซ้ำทีละย่าน → ทั้งเชียงใหม่ → จังหวัดอื่น
```

## 8. แนวทาง/รูปแบบการทำธุรกิจ (Business Approaches & Growth Motions)

### 8.1 โมเดลธุรกิจหลัก: B2B2C two-sided platform
- เก็บเงินจาก **ร้าน** (ขา 1–3) และ **พาร์ตเนอร์ข้อมูล/จอง** (ขา 4) — **ผู้ใช้ใช้ฟรี**
- ขายร้านแบบ **product-led + sales-assisted:** เริ่มฟรี ลองเอง แล้วค่อย upsell เป็น Starter/Growth/Pro เมื่อเห็นผล

### 8.2 ท่าการเติบโต (Growth motions)
1. **Land district-by-district (ปูพรมทีละย่าน):** ทำย่านให้ "หนาแน่นจนคุ้ม" ก่อนค่อยขยาย — ดีกว่าโปรยทั้งเมืองแบบบาง (แผนที่ว่าง = ตาย)
2. **Quest network เป็นเครื่องมือ growth ในตัว:** ร้านชวนร้าน, ผู้ใช้ชวนผู้ใช้
3. **Field-agent-led expansion:** ใช้ทีมภาคพื้นแบบผันแปรเปิดย่าน/เมืองใหม่
4. **Partnership-led demand:** โรงแรม/Klook/ททท./งานเทศกาล ป้อนผู้ใช้เป็นช่วงๆ
5. **Multi-city / multi-tenant ตั้งแต่ออกแบบ:** ระบบรองรับหลายเมืองแล้ว (scope-in-JWT) → ขยายไปจังหวัดอื่นโดยไม่ต้องเขียนใหม่ (ดู evolution path P0→P3 ใน memory/[05_stack_decision.md](05_stack_decision.md))

### 8.3 ปรัชญาเงินสองสกุล (ลดความเสี่ยงกฎหมายการเงิน)
- **Sparks** = แต้มฟรี เล่นเกมได้เต็มที่ (สุ่ม/เซอร์ไพรส์/ถี่ได้) เพราะไม่มีภาระเงินจริง
- **Coins** = มูลค่าจริง ค้ำด้วยเงินร้านที่เติมล่วงหน้า → ต้อง deterministic, ผ่าน money-gate, **ห้ามถอนเป็นเงินสด** และ **ห้ามแลกที่ร้านที่ออกทุนเอง** (เส้นแบ่ง e-money) → กันทั้งโกงและความเสี่ยงโดนตีความเป็นเงินอิเล็กทรอนิกส์/พนัน

### 8.4 สิ่งที่จะ "ไม่ทำ" (โฟกัส)
- ❌ ไม่แข่งเดลิเวอรี่/ฟู้ดออเดอร์กับ Grab/Wongnai
- ❌ ไม่สำรองจ่ายรางวัลแทนร้าน (no float risk)
- ❌ ไม่ทำเงินสกุลถอนได้ (เลี่ยง BoT) / ไม่ทำกลไกเสี่ยงโชคกับของมีมูลค่าจริง (เลี่ยง ม.8 การพนัน)

## 10. แผนสู่ตลาด & ก้าวสำคัญ (Milestones)

> อิงแผน build 5 เฟสใน [09_build_roadmap.md](09_build_roadmap.md) — ~26–30 สัปดาห์ถึง pilot นิมมาน

| P0 Foundation | ระบบ + กฎหมาย (จ้างทนายตั้งแต่ W0) | สเปกพร้อม |

## 11. ความเสี่ยง & สิ่งที่ต้องตัดสินใจ

### 11.1 ความเสี่ยงเชิงธุรกิจ
- **Incrementality ต่ำกว่า ~46%** → ร้านขาดทุน → เลิกใช้ → *ความเสี่ยงอันดับ 1* (คุมด้วยเพดานต้นทุนรางวัล + วัดผลจริงก่อนตั้งราคา)
- **Cold-start ไก่กับไข่** → คุมด้วยกลยุทธ์ supply-first ปูพรมทีละย่าน
- **ร้านไม่ยอมจ่าย** → ต้องพิสูจน์ "คนเดินเข้าร้านเพิ่มจริง" ใน pilot ก่อนคิดเงิน

### 11.2 ความเสี่ยงด้านกฎหมาย/กำกับ (ต้องให้ทนายไทยฟันธงก่อน go-live)
- **e-money / ธปท.:** Coins ต้องไม่ถูกตีความเป็นเงินอิเล็กทรอนิกส์ + escrow ไม่ใช่การรับฝากเงิน (เส้นแบ่ง = ห้ามแลกกับตัวเอง, ถอนเงินสดไม่ได้, ไม่โชว์เป็นบาทฝั่งผู้ใช้)
- **PDPA:** เก็บตำแหน่ง/ข้อมูลต้องมี DPA + consent ก่อนเก็บพิกัดใดๆ (รวมตอน agent เดินเก็บข้อมูล)
- **การพนัน ม.8/สลาก:** กลไกสุ่ม/แข่งความเร็ว/แจกของมีมูลค่า ต้อง deterministic — ดู [08_engagement_social_layer.md](08_engagement_social_layer.md) (C11–C14)
- **การจ้าง Field Agent:** จัดสถานะ freelancer + หัก ณ ที่จ่าย 3% (ค่าเริ่มต้น)

> รายการ "ต้องตัดสินใจ/เซ็นรับรอง" ครบอยู่ใน [OPEN_DECISIONS.md](OPEN_DECISIONS.md) (กลุ่ม B = ตั้งค่าพารามิเตอร์, C = กฎหมายต้องเซ็น)

## 12. ตัวชี้วัดที่ต้องดู (KPIs)

**ฝั่ง Supply:** จำนวนร้าน/ย่าน, ความหนาแน่น (ร้านในรัศมีเดิน), อัตราต่ออายุสมาชิก, งบ escrow ที่ร้านเติม
**ฝั่ง Demand:** ผู้ใช้ active, quest เริ่ม/quest จบ, อัตราจบ quest, รีวิว/ผู้ใช้
**เศรษฐศาสตร์หัวใจ:** **incrementality %** (ลูกค้าใหม่จริงจาก quest), ต้นทุนรางวัลต่อการแลก, take-rate revenue, CAC ร้าน, LTV ร้าน
**ความมั่นคงระบบ:** solvency recon = pass (เงินค้ำ Coin ครบเสมอ), เคสโกง

## 13. ขั้นต่อไป (Next steps สำหรับผู้ก่อตั้ง)

1. **จ้างทนายไทย "วันนี้"** — เริ่มเรื่อง e-money/escrow + PDPA + ม.8 (คอขวด 6–10 สัปดาห์)
2. **ตั้งค่าเพดานต้นทุนรางวัล** (B1/B2 ใน OPEN_DECISIONS — ระบบ fail-closed จนกว่าจะตั้ง)
3. **เริ่ม founder-led sales ที่นิมมาน** — ปิดดีลร้านชุดแรกให้หนาแน่น (พิสูจน์ความเต็มใจจ่าย)
4. **เปิดบัญชี cloud (Supabase/host) + deploy** ตาม [DEPLOY.md] เพื่อ pilot จริง
5. **รับ Field Agent ชุดแรก** (มช./พายัพ) เก็บข้อมูล + ชวนร้าน
6. **วัด incrementality ใน pilot** ก่อนตั้งราคาเต็ม — ถ้าไม่ถึง ~46% อย่าเร่ง scale
